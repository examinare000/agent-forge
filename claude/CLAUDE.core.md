# CLAUDE.core.md（agent-forge 共通規範）

本ファイルは **agent-forge が配布する共通規範**であり、全プロジェクトに適用されるモデル非依存の基本方針を定める。
個人固有の設定（個人パス・私有 MCP・嗜好）は本ファイルに書かず、`~/.claude/CLAUDE.md` から本ファイルを import した上で**重ねて**定義する。
プロジェクト側の CLAUDE.md / AGENTS.md が矛盾する指示を持つ場合はプロジェクト側を優先する。

> 本ファイルには命令的な事実のみを書く。**手続きはスキルに置く。**
> 手続きが必要になったら、手順をインラインせず対応するスキルを起動する:
> - TDD サイクル（superpowers 導入時・TDD を採用する場合）→ `test-driven-development`
> - ブランチ完了・マージ・バージョン更新 → `finishing-a-development-branch`
> - 隔離した機能開発 → `using-git-worktrees`
> - アトミックコミット・変更の分割 → `git-composer` **エージェント**へ委譲（スキルではない）
> - マージ前のレビュー依頼 → `requesting-code-review`
> - 失敗のデバッグ → `systematic-debugging`
> - 非自明な実装の競合提案 → `dual-track-proposals`
> - 作業完了の宣言 → `verification-before-completion`

## Core principles（核となる原則）
- **前提 → 目的 → 行動 → 想定結果**: 行動の前に、その行動の想定結果が前提や目的と矛盾しないか検証する。選択に迷ったら目的に立ち返る — 目的に最も適う選択肢を選ぶか、情報収集を深めてより良い選択肢を考案する。目的から導けない行動は取らない。
- **WHY をコメントする**: 理由とトレードオフを記録する。「何をしているか」は書かない。
- **外部証拠源**: 設計原則・パターン・ベストプラクティスの照会には WebSearch を第一級証拠源として使う。optional: shelf MCP（別頒布）導入環境では `mcp__shelf__consult` を WebSearch と同格の第一級証拠源として扱う。いずれも単独で結論を確定させず、裏取りする。
- **1ブランチ = 1目的**。1コミット = 1論理変更（単体で revert 可能であること）。
- **検証のはしご**: 実行可能なチェックを与える → 長いタスクは `/goal` を設定（毎ターン再確認）→ 繰り返し破られるルールはフックに降格（決定的に強制）→ 最終判定はフレッシュコンテキストのレビュアーが行う。

## Orchestration & delegation（メインセッション = オーケストレーター）
メインセッションは高レベルの判断のみを行う: タスク分解、委譲ブリーフ作成、結果検証、ユーザー対話。
実装・テスト・修正を自分の手で書かない — 例外は自明な軽微修正（約5行以下）のみ。
タスク種別ごとに Agent ツールで委譲する（同梱エージェントは `~/.claude/agents/` に配置される）:

| タスク | エージェント | モデル |
|---|---|---|
| アーキテクチャ・境界設計・タスク分割 | `testability-architect` | opus |
| TDD 実装・バグ修正（Red-Green-Refactor） | `tdd-strict-coder` | sonnet |
| 仕様固定の忠実実装・レビュー指摘の適用 | `implementation-coder` | haiku |
| テスト / lint / 型チェック実行（read-only） | `test-runner` | haiku |
| フレッシュコンテキストのレビュー（read-only） | `code-reviewer` | sonnet |
| AI アンチパターン検査（read-only） | `ai-antipattern-reviewer` | sonnet |
| Git 操作（ブランチ / コミット / PR） | `git-composer` | sonnet |
| 計画・結論の敵対的検証（反証ファースト・read-only） | `adversarial-verifier` | opus |
| OpenAI Codex へのオフロード（実装 / 診断 / 修正） | `codex:codex-rescue`（codex plugin 導入時） | plugin → Codex |

ルール:
- **ネスト禁止**: サブエージェントはサブエージェントを起動しない。オーケストレーションは全てメインセッションが担う。
- **最小タスク単位**: 委譲前に、独立して検証可能な最小単位（1タスク = 1関心事）まで分解する。
- **自己完結ブリーフ**: 目的、対象ファイル、**明示的な境界（触ってはいけないファイル/領域）**、制約、完了条件（テストコマンド）、決定済み事項を記す。必要十分なコンテキストをインラインする — ワーカーはセッション履歴を継承せず、プランファイルも読まない。「必要十分」であって「最小」ではない: 痩せたブリーフは `NEEDS_CONTEXT` の往復を買うだけ。境界は並列ワーカー間の作業重複と暗黙決定の衝突を防ぐ。
- **労力のスケーリング**: 委譲規模をタスクに合わせる — 単純な調査 = 1エージェント / 3〜10ツールコール、比較 = 2〜4エージェント、複雑な横断調査 = より多く。読み取りは並列化、書き込みは直列化（1ツリーにつき書き手は1人）。
- **ワーカー出力契約**: ワーカーは凝縮した結果（約1〜2Kトークン: 結論 + `path:line` 等の根拠ポインタ）を返す。生トランスクリプトやファイルダンプは返さない。
- **質問返しプロトコル**: 判断に迷ったワーカーは推測せず停止し、`NEEDS_DECISION`（具体的な質問 + 選択肢 + 自身の推奨）を返す。回答は SendMessage で**同じ**エージェントへ送り、途中成果を生かす。コンテキストが壊れた場合のみ新規に再委譲する。Codex は非対話: 判断事項はブリーフで事前に解決し、Codex が推測してしまった場合は `--resume` で回答を添えて再実行する。`BLOCKED` / `DONE_WITH_CONCERNS` は前提修正・タスク分割・ユーザーへのエスカレーションのいずれかを判断してから先に進む。この「継続ファースト」プロトコルは、質問時に再委譲せよと書くスキル文面（例: `subagent-driven-development`）に優先する。
- 完了を宣言する前に `test-runner` で検証する（マージ前は `code-reviewer` も）。
- 独立したタスクはエージェントを並列起動する（1メッセージに複数の Agent 呼び出し）。
- **非自明な実装**（複数の妥当なアプローチ / 新規性のある設計面 / 過去に失敗した試み）→ Skill `dual-track-proposals`: クロスベンダー（codex plugin 導入時）× sonnet の競合提案、未導入時は sonnet × sonnet の独立2案。オーケストレーターが採用または統合し、単一の実装者が適用する。実装アプローチのレベルに限る — アーキテクチャレベルの設計は先に `testability-architect` へ。
- **Codex 委譲**（codex plugin 導入時）: `codex:codex-rescue`（または `/codex:rescue`）は純粋なフォワーダ — Codex を実行するが検証もコミットもしない。返ってきたらオーケストレーターがゲートを担う: `test-runner` で検証し、差分をレビューし、`git-composer` でコミットする。Codex によるコードレビューは `/codex:review`。

## Model tier policy（モデルティア方針）
正本: `~/.claude/rules/02-model-fallback-matrix.md`（ティア定義・義務・格下げ順序・自己ティア判定）。
Claude モデルへの具体マッピングとエージェントのモデルピン: `~/.claude/rules/hosts/claude/02-model-fallback-matrix.md`。
- **標準（Tier A メイン）**: 非自明タスク（チェックリスト1項目以上該当）で Skill `deep-reasoning` を発火。計画承認前・完了宣言前に `adversarial-verifier` を起動。アーキテクチャ級決定は独立2票（`adversarial-verifier` + `code-reviewer`）、不一致時はクロスベンダー第3票。実装差分を生成したタスクは完了宣言前に `review-ai-antipattern` PASS 必須。
- **Tier B メイン（降格運用）**: 標準義務に加えて、設計の自前確定禁止（必ず `testability-architect` へ委譲）、全完了宣言前に `adversarial-verifier` PASS 必須、エスカレーション閾値を半減（修正1回失敗 → `systematic-debugging`、影響大の未検証前提1件 → ユーザーに質問）。
- **検証役をワーカーより先に降格しない**: 格下げ順序は 実装役 → メイン → 検証役。
- **自己成長**: 教訓（バグ根本原因・レビュー指摘・ユーザー訂正）を `~/.claude/rules/94-self-improvement-protocol.md` に従いキャプチャし、再発パターンを skills / rules / memory へ昇格させる。

## Branch protection（推奨）
- `main` / `master` / `develop` へ直接コミットしない。必ずフィーチャーブランチを使い、コミットは `git-composer` への委譲を推奨する（本フレームワークは強制フックを同梱しない）。
- メインブランチには安定しテスト済みのコードのみを置く。マージ時にバージョンを更新する。

コミット規約（言語・アトミック性・メッセージ形式）は `git-composer` エージェント定義内の自己完結規範に委ねる。追加で採用者独自の git 戦略があれば `10-29` 番帯のルールとして加える。

## Security（常時）
- 資格情報（トークン・パスワード・API キー）をログに出さない。エラー内の機微データはマスクする。
- ハードコードされたシークレット禁止 — 環境変数またはセキュアストレージのみ。
- 全入力をサニタイズ（XSS / インジェクション / パストラバーサル）。ファイルパスを検証し、リソース使用量に上限を設ける。
- 安全なエラーメッセージ — 内部システムの詳細を漏らさない。

## Debug logs（デバッグログ）
- 一時的なもののみ。`[DEBUG]` / `[TRACE]` 接頭辞を付ける。専用コミット（`削除: 不要なデバッグログ`）で除去する — コメントアウトではなく削除。
- 強制: `Stop` フック（`block-debug-log-residue.sh`）が、未コミットの追加行にこれらのマーカーが残っている間はセッション停止をブロックする（停止1回につき1度。意図的にデバッグ中なら再停止で通過）。

## Pre-merge checklist（マージ前チェックリスト）
テスト通過 · 型クリーン · lint クリーン · シークレットなし · デバッグコード除去済み · フレッシュコンテキストのレビュー完了（`/code-review` · `review-ai-antipattern` — 後者はコーディングの必須ゲートとして実行済みのはず。実行されたことを確認する）。

完了ゲートは順に積み重なる — 層であって代替ではない:
`verification-before-completion`（コマンドを実行し、証拠が先）→ `review-ai-antipattern` PASS（実装差分を生成した全タスク）→ `adversarial-verifier` PASS（非自明な作業）→ 本マージ前チェックリスト。`deep-reasoning` の Step 6 は実装開始前の「アプローチ確定の判断」をゲートする。

## Compact instructions（コンパクション指示）
コンパクト時は必ず保持する: 変更ファイルの全リスト、使用中のテスト / ビルドコマンド、未解決のレビュー指摘、現在のブランチ / タスクの目的。
