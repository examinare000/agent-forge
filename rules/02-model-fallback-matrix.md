# 02. モデルティア運用マトリクス（全ホスト共通・正本）

メインセッションを担うモデルのティアに応じて、役割分担と義務を切り替えるための**正本**。
標準運用は **Tier A メイン**。Tier B 以下での運用では、単独判断を制限し、
**構造（発火義務・検証強化・独立多数決）で常時補う**。

> **単一更新点**: 本ファイルはティアを**抽象定義**でのみ扱う。各ホスト（Claude Code / Codex / Gemini）の
> 具体的なモデル名対応は `hosts/<host>/02-*.md`（Claude は `hosts/claude/02-model-fallback-matrix.md`）が持つ。
> 他のルール・スキルの本文はティア名（A/B/C）でのみ参照する（description 等の自然文での言い換えは検索性のため許容）。

## ティア定義（ベンダー中立）

| ティア | 定義 | 位置づけ |
|---|---|---|
| **Tier A** | 各ベンダーの**同世代フラッグシップ推論モデル**（最大 reasoning effort 設定時） | **標準運用のメイン**（義務定義の基準ティア）。判断力は高いが、推論の明示化と独立検証を常時義務付ける |
| **Tier B** | 各ベンダーの標準・中位モデル | 降格運用（Tier A 不在時）。単独判断を制限し検証を最大化する。ワーカー用途では標準構成の一部 |
| **Tier C** | 軽量・高速系モデル（mini / nano / flash 級） | 委譲・リレー・機械的作業専用。**判断役（設計・レビュー・オーケストレーター）禁止** |

## クロスベンダーのティア対応（Codex / Gemini）

**正本規則（抽象）**: 各ベンダーの**同世代フラッグシップ推論モデル（最大 reasoning effort 設定時）= Tier A 相当**。標準・中位モデル = Tier B 相当。軽量・高速系（mini / nano / flash 級）= Tier C 相当。

**非正本の例示表**（以下は参考例示であり、モデル改名時に本表の更新を必須としない。抽象規則が正本）:
- **GPT 系**: codex 系フラッグシップ（reasoning effort: high）→ Tier A 相当 / 同系の標準 effort・中位モデル → Tier B 相当 / mini・nano 級 → Tier C 相当
- **Gemini 系**: Pro 世代最上位 → Tier A 相当、Flash 系 → Tier B～C 相当

**義務の代替**: 非 Claude 環境で Tier 義務が参照するエージェント（`adversarial-verifier` 等）の起動経路が無い場合（Gemini 単独セッション等）は、クロスベンダー MCP 相互検証（gemini-cli ⇄ codex）または**ユーザー承認**で代替する（実行不能な義務を課さない）。Codex 単独セッションは `.codex/agents/` の同名エージェントで義務を履行する。

## 標準運用の義務（Tier A）と降格時の追加義務（Tier B）

**非自明タスクの定義**: Skill `deep-reasoning` の発火チェックリストに 1 項目以上該当するタスク。
発火閾値は**全メイン共通で 1 項目以上**。

- **標準運用（Tier A メイン）**:
  - 非自明タスクで Skill `deep-reasoning` を必須発火。
  - 計画承認・完了宣言前に `adversarial-verifier` を起動。
  - **アーキテクチャ級決定**（アーキテクチャ・データモデル・公開 API・非可逆操作）は独立 2 票
    （`adversarial-verifier` + `code-reviewer`）を必須とし、不一致時はクロスベンダー第3票
    （手順の正本: `03-agent-behavior.md`「委譲と検証の原則」）。
  - **実装差分を生成したタスク**は、完了宣言前に Skill `review-ai-antipattern` を必須実行する
    （重大指摘は修正してから完了宣言。教訓の記録は `94-self-improvement-protocol.md`）。
- **Tier B メイン — 標準義務に加えて**:
  - 設計判断を自前で確定しない。必ず `testability-architect` へ委譲するか、`03-agent-behavior.md` の多数決手順にかける。
  - 完了宣言前に `adversarial-verifier` の PASS を必須とする。
  - `03-agent-behavior.md` のエスカレーション閾値を**強化**（修正 2 回失敗 ➔ 1 回失敗で `systematic-debugging`、影響大の未検証前提 3 件 ➔ 1 件で質問）。
  - 楽観的表現（「これで完璧」「問題ないはず」「おそらく動く」）の出力禁止。完了・成功の主張には検証コマンドと結果の添付を義務付ける。

## 格下げの順序（検証役は最後まで守る）

コスト等の理由でエージェントのモデルピンを下げる場合、順序は **実装役 ➔ メイン ➔ 検証役**。
検証役（`testability-architect` / `adversarial-verifier`）の Tier A ピンは、Tier A モデルが
（サブエージェント用途を含め）存在する限り**維持する**。
「メインより強い検証者」の非対称構造が品質担保の核であり、検証役を先に下げると全体が崩れる。
検証役を Tier B に降格せざるを得ない場合は、代償としてクロスベンダー多数決（第3票）を**常時必須**に切り替える。
ホスト別の具体的な降格手順は `hosts/<host>/02-*.md` を参照。

## 自己ティア判定（セッション開始時・モデル変更時）

1. 自分のモデル名を確認する（環境情報 or 自己認識）。ティアへの対応は `hosts/<host>/02-*.md` の対応表による。
2. **標準ティア（Tier A）でなければ**、作業開始前に適用ティアを 1 行宣言する。標準運用時は宣言不要。
   例: 「**Tier B 運用**: 設計自前確定禁止・完了宣言前 adversarial-verifier PASS 必須を適用します」
3. 自モデルのティアを判定できない場合は **Tier B の義務を適用する**（安全側デフォルト）。
4. 宣言したティアの義務（上記）を、そのセッション中は解除しない。
   非 Claude 例: 「**Tier B 運用（Gemini）**: 設計委譲 + クロスベンダー多数決を適用」
