# 30. ドキュメント管理

## ドキュメント形式ポリシー（正本の所在）🟠

本プロジェクトは **Agentic Coding** を基本とするため、ドキュメントは
**AI が理解・運用しやすいこと**と**人間にとって可読性が高いこと**の両面で最適化する。
判断軸は「**正本 vs 生成物**」と「**常時ロード vs オンデマンド**」。

> **基本原則: Markdown を単一正本とし、人間向け HTML は SSG（MkDocs）で生成する。**
> Agentic Coding のベストプラクティス（単一正本・段階的開示・低トークン）に従う。
> 背景・調査はプロジェクトの `docs/research-reports/` 配下に残す。

| 区分 | 正本（編集対象） | 役割 |
|------|------|------|
| 設計書・仕様書 | **Markdown**（`docs/design/*.md`） | AI が低トークンで読め、git 差分・レビューも軽い単一正本。図は **Mermaid**。HTML は `mkdocs build` で生成（手編集しない） |
| ADR・要件・計画・運用 | **Markdown**（`docs/adr/`・`prd/`・`plan/`・`operations/`・`tech/`） | 意思決定・現在状態の **AI コンテキスト圧縮層**。コードベース全調査を不要化する |
| 索引（段階的開示） | **Markdown**（`docs/README.md`・ルート `AGENTS.md`） | 「何を・いつ読むか」の索引。本体は必要時にオンデマンド取得 |
| 人間向け表示 | **HTML（生成物）** | `docs/design/*.md` から SSG で生成。手で編集しない。`docs/design/app-ui.html` 等の手書きビジュアル正本は例外として残置 |

- **正本は Markdown**: 設計・仕様も Markdown を唯一の正とする。HTML を手保守の正本にしない（二重管理・乖離・トークン肥大を招くため）。
- **HTML は生成物**: 人間向けの体裁は `mkdocs build` で生成する。生成物（`site/`）は編集せず、コミットもしない。
- **図は Mermaid**: 手書き SVG ではなく Mermaid（テキスト・差分追跡可）を用いる。
- **実装の正本はコード**: 設計書は責務・契約・意図の記述に徹し、実装コードは掲載しない。

## ディレクトリ構造
- `docs/README.md`: **ドキュメント索引**（段階的開示の入口）
- `docs/design/`: **設計書・仕様書の正本（Markdown）**。HTML は生成物
- `docs/prd/`: 要件定義 (PRD)
- `docs/adr/`: アーキテクチャ決定レコード (ADR)
- `docs/plan/`: 実装計画・タスク指示書（作業領域。完了後は恒久ドキュメントへ移し削除）
- `docs/tech/`: 技術文書（DBスキーマ、API設計等）
- `docs/operations/`: 運用文書
- `docs/research-reports/`: 調査・評価レポート

## ドキュメント規約

### PRD
- **配置**: `docs/prd/<機能域>-<内容>.md`
- **構成案**: 概要（目的と背景） / 機能・非機能要件 / 受け入れ基準（条件・実行・結果）

### ADR
- **配置**: `docs/adr/<3桁連番>-<決定内容>.md`
- **ポリシー**: 既存ADRは変更せず、新ADRで上書き（ステータス: 採用済み / 廃止 / 検討中）
- **構成案**: 背景 / 検討した選択肢（メリット・デメリット） / 決定 / 理由 / 結果（影響）

### 技術文書
- **配置**: `docs/tech/<分野>-<内容>.md`

## 運用＆ライフサイクル

### 更新タイミング
- 設計書・仕様書（HTML 正本）: 仕様・設計の確定変更時に同期
- PRD: 要件変更時に即座
- ADR: 技術決定時に新規作成
- 技術文書・運用文書: 実装/運用変更時に即座に同期

### 品質基準
- 対象読者の明確化、日本語記述、最終更新日の記載。

### 作成原則
- **プランニング時のコードスニペット禁止**: 計画・設計段階の文書では過度なコードスニペットを記載しない。意図・方針・インターフェースを自然言語で記述し、実装の詳細はコード本体に委ねる（コードと文書の二重管理・乖離を防ぐため）。例示が不可欠な場合のみ最小限に留める。
- **アスキーアート禁止**: 図示が必要な場合はアスキーアート（ASCII art）を使わず、Mermaid 等のテキストベース作図記法を用いる（保守性・可読性・差分追跡性のため）。ファイル構成・ディレクトリツリーも罫線（`├──` 等）で描かず、表または箇条書きで表現する。
- **設計書（HTML 含む）も同原則を適用**: `docs/design/` 配下の設計書（`*.html` のビジュアル設計書を含む）も上記 2 原則に従う。実装コード（クラス本体・関数実装・設定ファイル中身等）は掲載せず、責務・インターフェース・データ契約は自然言語・表・Mermaid／SVG で示す。API リクエスト/レスポンスや DB スキーマなど例示が不可欠な場合は、最小限の JSON 例またはフィールド表（項目・型・説明）に留める。実装の正本はコード、設計書は意図と契約の記述に徹する。

### エージェント行動指針
- **作成**: 明示的要求時のみ適切なテンプレートに従い作成。
- **参照**: 正確なパスで参照し、関連ドキュメントとの整合性を検証。
- **更新**: 影響範囲のドキュメントを同時に更新し、変更理由を明記。

### メンテナンス＆オンボーディング
- **確認**: 月次（乖離チェック）、四半期（体系見直し）、リリース前（更新確認）。
- **推奨読順**: `README.md` ➔ `docs/prd/` ➔ `docs/design/*.html`（設計・仕様の正本） ➔ `docs/adr/`
