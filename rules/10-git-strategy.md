# 10. Git戦略
ブランチ・コミット運用の正本（Git-Flowベース）。

## 必須ルール
- 1ブランチ = 1イニシアチブ（短命に）。
- `main` / `develop`への直接コミット、共有ブランチへの force-push 禁止。
- `git add .` 禁止（意図しない混入防止）。WIP連投・無関係な変更の混在禁止。
- **変更系 git/gh はメイン（オーケストレーター）で実行しない**。コミット・ブランチ削除/改名・マージ・rebase・reset・push・PR 等は必ず Agent ツールで `git-composer` を起動して委譲する。読み取り専用（`status`/`diff`/`log`/`branch --list`/`gh ... view|list`）のみメインで実行可。詳細は[フック整合](#フック整合メインとリポジトリ)。
  - **例外（メインで直接実行可）**: ブランチ**作成・切替**（`git switch [-c]` / `git checkout -b/-B` / `git branch <新規>`）はアトミック性・メッセージ規約の判断を伴わず、ブランチ保護フックも commit を伴わないため無関係。一行のブランチ操作のために git-composer を spawn する（数万トークン＋数十秒）のは純粋な浪費なので委譲しない。ただしブランチ削除/改名（`-d`/`-D`/`-m`）や作業ツリー破棄（`git checkout -- <path>`）は破壊的なので従来どおり委譲する。
- **1タスク = 1回の委譲にまとめる（多重起動禁止）**🟠。関連する一連の変更系操作（例: ステージ→アトミック commit→push→PR 作成→マージ→`main` 最新化）は、操作ごとに git-composer を起動せず、**1 回の git-composer 起動に全手順をまとめて渡して**実行させる。git-composer はサブエージェントなのでフック免除で内部の git/gh を自由に連続実行できる（[フック整合](#フック整合メインとリポジトリ) 参照）。操作単位の個別委譲は subagent 起動コスト（数万トークン・数十秒）を毎回支払う浪費であり禁止。（多レポ採用時）複数リポジトリにまたがる場合も、各リポの手順を 1 つの委譲プロンプトに列挙する（リポ単位で並列起動は可、操作単位の逐次起動は不可）。

## ブランチ構成
- **`main`**: 本番用。タグリリースのみ（直コミ禁止）。
- **`develop`**: 統合開発用（直コミ禁止）。
- **`feature/<task-name>`**: 新機能（developから分岐 ➔ developへマージ）。
- **`release/<version>`**: リリース準備（developから分岐 ➔ main + developへマージ）。
- **`hotfix/<name>`**: 緊急修正（mainから分岐 ➔ main + developへマージ）。
- ※命名例: `feature/user-auth-session-timeout`, `release/v0.5.0`, `hotfix/critical-sync-bug`

## アトミックコミット
1. **1コミット = 1論理変更**。
2. **複数ファイルは原則別コミット**（相互依存でビルドエラーになる場合のみ同一コミット）。
3. **独立した取り消し可能性**: 単体で revert 可能であること。

## コミットメッセージ規約
- **形式**: `<種別>: <説明>`（日本語、1〜2文、WHY優先）。
- **種別**: 機能, 修正, 改善, 削除, 移動, 名前変更, テスト, 文書, 設定, 構成, 統合
- **禁止語**: `update`, `WIP` は使用禁止。
- **例**: `改善: ボタン色をCSS変数に移行（保守性向上）` / `統合: feature/user-authをdevelopにマージ`

## マージ戦略
- マージコミット（`git merge --no-ff` 等）許容。
- コンフリクト解決後は `統合: <理由>` でコミット。

## フック整合（メインとリポジトリ）
以下の 2 つの PreToolUse フックが規約をコマンド実行前に強制する。**フックの実体はユーザースコープ（`~/.claude/settings.json` + `~/.claude/hooks/`）で配線されており、本テンプレートには同梱されない**。フック未配線の環境（テンプレート配布先等）では、本節の規約を規律として遵守すること。**フックが配線されている場合、それを修正・無効化してはならない**（セキュリティ／規約ガードレール）。誤検知に見えても、まず下記の正しい手順に合わせる。

1. **変更系 git/gh はメインで不可（`delegate-git-to-composer.sh`）**
   - メイン（オーケストレーター）が `commit`/`push`/`pull`/`merge`/`rebase`/`reset`/`revert`/`cherry-pick`/`branch -d/-m`/`checkout -- <path>`/`tag 作成`/`gh pr create|merge` などの**変更系**を実行するとブロックされる（サブエージェントは免除）。
   - **ブランチ作成・切替は例外でメイン実行可**: `git switch [-c]` / `git checkout -b/-B` / `git branch <新規>` はブロックされない（判断を伴わない・commit を伴わず保護フックと無関係なため、spawn コストの方が高くつく）。
   - 対応: 上記**変更系**は最初から **`git-composer` に委譲**する（メインで試行しない）。読み取り専用 git/gh とブランチ作成・切替はメインで実行してよい。
   - **委譲は一連の手順をまとめて 1 回**。フックは「個別操作のブロック」だが、回避策として操作ごとに git-composer を起動してはならない。1 タスク分の変更系手順（commit〜push〜PR〜マージ〜main 最新化）を 1 つの委譲プロンプトに列挙し、git-composer に連続実行させる（多重起動はトークン・時間の浪費。必須ルール参照）。

2. **保護ブランチ判定は「コミット対象ディレクトリ」のブランチを見る（`block-protected-branch-commit.sh`）**
   - フックはペイロードの `cwd`（永続シェルの cwd）を基準に、`cd <repo>`／`git -C <repo>` も解釈して **実際にコミットされるリポのブランチ**を `git -C <対象dir> rev-parse` で評価する。よって `cd <repo> && git commit ...` も対象リポのブランチで正しく判定される（別リポ/サブモジュール採用時も対応済み）。
   - 推奨手順（堅牢性のため依然有効）: **コミット対象リポへは独立したステップで `cd` し**、`git branch --show-current` で目的のフィーチャーブランチを確認してからコミットする。セッションの cwd が別リポ（例: `main` 上の親リポ）にあっても、対象リポが feature ブランチなら通る。
   - 注意: （別リポ/サブモジュール採用時）親リポが `main` でも、別リポの feature ブランチへのコミットは誤ブロックされない。一方、対象リポ自身が `main`/`master`/`develop` なら従来どおりブロックされる。

## （任意）Git Submodule / マルチレポ運用ルール
> **適用条件**: プロジェクトが Submodule / マルチレポ構成を採用する場合のみ。**単一リポ構成では本節は非適用**（リポルート直下で通常どおり作業する）。各プロジェクトの実構成は、そのプロジェクト固有の規約ドキュメント（例: `docs/project-conventions.md`）を正とする。

Submodule 採用時、各コンポーネント（例: `<submodule>/`）は個別のリポジトリとして扱う。

1. **サブモジュール内での開発**:
   - 各サブモジュール（例: `<submodule>/`）内でブランチを作成し、アトミックコミットを行う。
   - 変更が完了したらサブモジュールのリモートへ push する。
2. **親リポジトリでのポインタ更新**:
   - サブモジュール内のコミットを親リポジトリに反映させるため、親リポジトリ側で `git add <サブモジュール名>` し、ポインタ（コミットハッシュ）の更新をコミットする。
   - コミットメッセージ例: `構成: <submodule> サブモジュールを最新の <hash> に更新`
3. **同期の維持**:
   - 作業開始前に必ず `git submodule update --remote --recursive` 等で最新状態に同期すること。

## （任意）典型フロー（サブモジュール開発）
> 単一リポ構成では不要。下記のうち **commit/push 等の変更系は `git-composer` が実行**する（メインは委譲のみ）。`git checkout -b`（ブランチ作成）はメインで直接実行してもよい。各リポジトリへは**独立ステップで `cd`** し、`cd` を `git commit` と同一行に詰めない（[フック整合](#フック整合メインとリポジトリ) 2.）。`git add .` は使わずファイルを明示する。

```bash
# 1. サブモジュール内で開発（cwd を対象サブモジュールに固定してから操作）
cd <submodule>
git checkout -b feature/new-logic          # フィーチャーブランチ
# 実装・テスト
git add path/to/changed_file               # 明示追加（git add . 禁止）
git commit -m "機能: ロジック追加"          # cwd=<submodule>・feature ブランチで実行
git push -u origin feature/new-logic

# 2. 親リポジトリでポインタ更新（cwd を親へ移してから）
cd /path/to/repo-root                       # 単独ステップで cwd を親に切替
git branch --show-current                   # 親が feature ブランチであることを確認
git add <submodule>                         # サブモジュールのポインタのみ
git commit -m "構成: <submodule> サブモジュールを <hash> に更新"
```
